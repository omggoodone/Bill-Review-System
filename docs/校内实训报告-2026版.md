## 1. 项目概述

### 1.1 项目背景

随着组织规模的扩大，传统的纸质票据审批流程面临效率低下、审批周期长、数据难以追溯和统计等问题。为解决这些痛点，本项目开发了一套在线票据审核系统，实现了从票据提交、审核流转到数据统计的全程数字化管理。

### 1.2 项目目标

构建一个支持四角色协作的在线票据审核系统：

- **超级管理员**：全局数据监控、审核员工作量统计、创建管理员账号
- **系统管理员**：用户管理（增删停用）、票据类别管理、注册审批
- **票据审核员**：审核票据（通过/退回+意见）、积压预警、批量审核
- **普通用户**：票据 CRUD、提交审核、查看审核结果

### 1.3 票据流转

```
草稿(0) → 已提交(1) → 已通过(2)  
                    → 已退回(3) → 可重新编辑提交
```

### 1.4 系统角色

| 角色 | role_key | 登录入口 | 职责 |
|------|----------|----------|------|
| 超级管理员 | admin | /admin/super | 全局数据监控、审核员工作量、创建管理员 |
| 系统管理员 | admin_user | /admin/dashboard | 用户/类别管理、注册审批、概览统计 |
| 票据审核员 | reviewer | /bill/manage | 审核票据（通过/退回+意见）、批量审核 |
| 普通用户 | user | /bill/manage | 票据 CRUD、提交、查看审核结果 |

### 4.4 安全设计

- **认证**：JWT Token 30分钟过期，存储在 Redis，每次请求通过 Filter 验证
- **鉴权**：三级控制——`router.beforeEach`（路由） + `v-hasPermi`（按钮） + `@PreAuthorize`（接口）
- **单点登录**：登录时清除旧 Token，旧设备下次请求返回 401
- **强制下线**：停用/删除用户时清除 Redis Token
- **数据隔离**：普通用户只能查看自己的票据（`create_by` 过滤），审核员只能看到活跃用户票据（防御性过滤）

### 4.5 Redis 缓存体系

| Key 模式 | 用途 | 过期 |
|------|------|:--:|
| login_tokens:{uuid} | 登录态 | 30分钟 |
| login_user_ids:{userId} | 强制下线映射 | 30分钟 |
| captcha_codes:{uuid} | 验证码 | 2分钟 |
| sys_dict:{type} | 字典缓存 | 永久 |
| sys_config:{key} | 系统配置缓存 | 永久 |
| pwd_err_cnt:{username} | 密码错误计数 | 10分钟 |

## 5. 核心功能实现

### 5.1 认证鉴权（JWT + Redis）

用户登录时，`SysLoginService` 验证密码后调用 `TokenService.createToken()` 生成 UUID Token：

```java
public String createToken(LoginUser loginUser) {
    // 踢掉旧登录——单点登录
    String oldToken = redisCache.getCacheObject(getUserIdKey(loginUser.getUserId()));
    if (StringUtils.isNotEmpty(oldToken)) {
        redisCache.deleteObject(getTokenKey(oldToken));
    }
    String token = IdUtils.fastUUID();
    loginUser.setToken(token);
    refreshToken(loginUser); // 写入 Redis
    return createToken(claims);
}
```

每次请求经 `JwtAuthenticationTokenFilter` 从 Redis 查 Token。Redis 无数据时返回 401 "已在其他设备登录"。

### 5.2 票据 CRUD

核心实现位于 `BizBillServiceImpl`：

- **新增**：`addBill()` 支持保存草稿（临时编号 DRAFT-xxx）和直接提交（正式编号 BILL-YYYYMMDD-NNNN）
- **查询**：`queryBills()` 根据角色自动过滤——普通用户只看自己，审核员看活跃用户票据
- **编辑**：`updateBill()` 仅草稿/已退回状态可编辑，校验数据归属
- **删除**：`deleteBill()` 仅草稿可删除，同时清除附件 DB 记录和磁盘文件
- **提交**：`submitBill()` 草稿→已提交，生成正式编号

### 5.3 审核流程

`reviewBill()` 实现：

```java
public void reviewBill(BizBillReviewDto dto) {
    // 校验 action: 1-通过, 2-退回
    // 仅已提交状态可审核
    bill.setStatus("1".equals(dto.getAction()) ? "2" : "3");
    bill.setAuditBy(SecurityUtils.getUsername());
    // 记录审核历史到 biz_audit_log
    auditLogMapper.insert(log);
}
```

支持批量审核 `batchReview()`，返回成功数量和失败 ID 列表。

### 5.4 用户管理（软删除 + 强制下线）

```java
// 删除用户：软删除 + 清理非通过票据
billService.deleteNonApprovedBillsByUser(u.getUserName());
user.setDelFlag("2"); // 软删除
tokenService.deleteLoginUserByUserId(userId); // 强制下线
```

### 5.5 附件管理

- **上传**：前端 FileUpload 组件 → `POST /common/upload` → `FileUploadUtils` 校验大小/扩展名 → 存到磁盘
- **存储**：`saveAttachments()` 解析路径字符串 → 逐条写入 `biz_bill_file` 表
- **删除**：`deleteFilesAndRecords()` 删除 DB 记录同时清磁盘文件，防止磁盘孤文件
- **展示**：Spring `ResourcesConfig` 映射 `/profile/**` → 本地文件目录

### 5.6 系统初始化

首次访问时，`GET /api/init/status` 返回 `initialized: false`。前端显示初始化表单，输入邮箱 → `POST /api/init` 创建首个超管账号（发送邮件通知）。

## 6. 前端设计

### 6.1 路由与菜单

```
登录 → getInfo (获取 roles + permissions) → getRouters (获取菜单树)
  → filterAsyncRouter (component字符串→Vue组件) → router.addRoute
  → 按角色跳转首页
```

动态路由通过 `import.meta.glob('./../../views/**/*.vue')` 匹配所有视图文件，后端返回菜单 JSON 的 `component` 字段与文件路径一一对应。

### 6.2 权限控制

```html
<!-- 路由级 -->
{ path: '/admin/super', permissions: ['biz:admin:list'] }

<!-- 按钮级 -->
<el-button v-hasPermi="['biz:bill:add']">新增</el-button>

<!-- 请求级 -->
@PreAuthorize("@ss.hasPermi('biz:bill:add')")
```

### 6.3 状态管理

- **user store**：Token、用户名、角色列表、权限列表
- **permission store**：路由表、侧边栏菜单
- **dict store**：字典数据本地缓存，自动按值匹配颜色

### 6.4 移动端适配

- 表格操作列 `fixed="right"`——横向滑动时按钮始终可见
- 工具栏按钮转圆形图标（隐藏文字）
- 搜索栏表单项堆叠
- 侧边栏覆盖模式 + 遮罩层
- 分页精简（隐藏跳转/条数选择）

## 7. 部署运维

### 7.1 部署环境

- **服务器**：WSL Ubuntu 22.04
- **Java**：OpenJDK 17
- **Nginx**：1.18.0（反向代理 + 静态文件）
- **MySQL**：8.0 + Redis 6.0

### 7.2 Nginx 配置要点

```nginx
# 前端静态文件
root /opt/bill-review/dist;
location / { try_files $uri /index.html; }

# API 转发到后端
location ^~ /prod-api/ {
    proxy_pass http://127.0.0.1:8080/;
}

# 静态文件缓存 + Gzip
location ~* \.(js|css|png|jpg)$ { expires 30d; }
gzip_static on;
```

### 7.3 部署流程

```bash
# 前端构建
cd zsc_frontend && npm run build:prod

# 后端构建
cd zsc_backend && mvn clean package -DskipTests

# 部署
cp -r dist/* /opt/bill-review/dist/
cp zsc-admin/target/zsc-admin.jar /opt/bill-review/
cd /opt/bill-review && java -jar zsc-admin.jar
```

### 7.4 操作日志

管理员和超管通过 `/monitor/operlog` 页面查看所有用户的操作记录（增删改查），支持按模块、操作人、类型、状态、时间筛选，支持清空和导出 Excel。
