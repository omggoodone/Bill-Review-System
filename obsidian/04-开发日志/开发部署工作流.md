# 开发部署工作流

## 全景视图

```
┌─────────────────────────────────────────────────────────────────┐
│                        完整工作流                                │
│                                                                 │
│  修改代码 → 构建 → 部署 → 启动服务 → 暴露公网 → 验证            │
│                                                                 │
│  ─────── 开发阶段 ───────  ────── 发布阶段 ──────               │
│     (频繁、快速反馈)          (阶段性、面向用户)                  │
└─────────────────────────────────────────────────────────────────┘
```

## 架构总览

```
                         修改源码

            ┌───────────────┴───────────────┐
```

## 阶段一：修改代码

### 改前端 → 快速验证

```
修改 .vue / .ts / .css 文件
        │
        ▼
  Vite dev server (npm run dev)
  浏览器实时热更新（毫秒级）
  适合：调样式、调交互、联调 API
```

**不经过**构建，不经过部署，直接看到效果。

### 改后端 → 调试验证

```
修改 .java 文件
        │
        ▼
  IDE 重启应用（或 mvn spring-boot:run）
  Postman / 浏览器验证 API
  适合：改业务逻辑、改数据库操作
```

**不经过**打包，不经过 Nginx，直接连 8080 端口。

### 开发模式的边界

| 能验证的 | 不能验证的 |
|----------|-----------|
| 代码逻辑是否正确 | 生产构建是否有问题 |
| API 返回是否符合预期 | Nginx 代理是否正常生效 |
| 页面样式是否 ok | Gzip 压缩效果 |
| — | 完整用户体验（加载速度等） |

> 规则：**日常开发用开发模式，阶段性里程碑用部署模式验收。**

## 阶段二：构建

### 为什么需要构建

开发模式下的代码是给人看的（`.vue` 单文件组件、`.java` 类文件），构建是把它们变成机器最优运行的形式：

| | 构建前 | 构建后 |
|------|--------|--------|
| **前端** | `.vue` / `.ts` / `.scss` 散文件 | `dist/` 下 HTML + 合并压缩的 JS/CSS |
| **后端** | `.java` 源码 + `pom.xml` | `zsc-admin.jar`（fat JAR，内嵌 Tomcat + 所有依赖） |

### 构建命令

```bash
# 前端：Vite 编译、打包、压缩、文件名加 hash
cd zsc_frontend
npm run build:prod
# 产物：dist/ （几百个文件 → 浏览器直接加载）

# 后端：Maven 编译、测试跳过、打包 fat JAR
cd zsc_backend
mvn clean package -DskipTests
# 产物：zsc-admin/target/zsc-admin.jar （一个文件包含一切）
```

### 构建的本质

```
源代码（开发态）              →        产物（运行态）

.vue 单文件组件                         .js 浏览器可直接执行的脚本
TypeScript 类型标注                     纯 JavaScript（类型被擦除）
SCSS 嵌套样式                           压缩合并的 CSS
import 模块引用                         合并后的 chunk 文件
.java 类文件                            编译后的 .class 字节码
pom.xml 依赖声明                        所有 jar 打入一个 fat JAR
```

> 构建 = 将开发态的表达，翻译为部署态的最小运行单元。

## 阶段三：部署

### 部署的目标

把构建产物放到运行环境（WSL/服务器），配置好 Nginx 和服务，让系统以**生产形态**运行。

### 部署步骤

```
1. 确保基础服务运行
   ├─ MySQL  (systemctl start mysql)
   ├─ Redis  (systemctl start redis-server)
   └─ 端口检查 (ss -tlnp)

2. 放置构建产物
   ├─ dist/*       → /opt/bill-review/dist/
   └─ zsc-admin.jar → /opt/bill-review/

3. 配置 Nginx
   ├─ 复制 nginx.conf → /etc/nginx/conf.d/bill.conf
   ├─ 移除默认站点（避免抢占80端口）
   └─ nginx -t && nginx -s reload

4. 启动后端
   └─ nohup java -jar zsc-admin.jar > logs/app.log 2>&1 &

5. 验证
   ├─ curl :8080（后端）
   └─ curl :80（前端）
```

### 修改后重部署

```
修改了前端源码：
  npm run build:prod  →  cp -r dist/* /opt/bill-review/dist/  →  完成（Nginx 无需重启）

修改了后端源码：
  mvn clean package  →  pkill -f zsc-admin.jar  →  cp jar 到 /opt/bill-review/
  →  nohup java -jar ... &  →  完成
```

| 修改内容 | 需要重做的事 | 是否停服 |
|----------|-------------|---------|
| 前端 .vue/.css | build + 拷贝 dist | 不停 |
| 前端路由/配置 | build + 拷贝 dist | 不停 |
| 后端 .java | package + 重启 JAR | 短暂中断（几秒） |
| 后端配置 (yml) | 重启 JAR | 短暂中断 |
| 数据库 (SQL) | 直接执行 | 不停 |

## 阶段四：暴露公网

### 为什么需要这一步

WSL/家庭网络的 IP 是内网地址（192.168.x.x），外网无法直接访问。需要一条**隧道**把公网请求引流到本地。

### 原理

```
公网用户 → https://xxx.ngrok-free.dev
                │
                ▼  ngrok 服务端（云端）
                │
                ▼  ngrok 客户端（你本机，与云端维持长连接）
                │
                ▼  localhost:80 (Nginx)
                │
                ▼  你的应用
```

ngrok 客户端 = 桥梁，维持一条从你电脑到 ngrok 云端的加密隧道。外部请求通过隧道"穿透"到你本地。

### 启动

```bash
ngrok http 80
```

- 控制台：http://localhost:4040（查看实时请求、重放、检查响应）
- 免费版限制：每次重启域名随机、有带宽限制

### 验证公网可达

```bash
curl -s -o /dev/null -w "%{http_code}" https://xxx.ngrok-free.dev/
# 期望输出：200
```

## 完整工作流总结

### 日常开发循环

```
修改代码 → 本地验证(npm run dev / IDE运行) → 改到满意
                                                │
                                                ▼
                                         需要公网演示/验收？
                                           /          \
                                         否            是
                                          │             │
                                          ▼             ▼
                                      完成       构建 → 部署 → ngrok暴露
                                                       │
                                                       ▼
                                                  公网验证
```

### 命令速查

```bash
# ===== 开发 =====
npm run dev                          # 前端开发模式
mvn spring-boot:run                  # 后端开发模式（或 IDE 直接跑）

# ===== 构建 =====
npm run build:prod                   # 前端构建
mvn clean package -DskipTests        # 后端构建

# ===== 部署 =====
# 首次部署
sudo mkdir -p /opt/bill-review/{dist,logs,uploads}
sudo cp -r dist/* /opt/bill-review/dist/
sudo cp zsc-admin.jar /opt/bill-review/
sudo cp deploy/nginx.conf /etc/nginx/conf.d/bill.conf
sudo rm /etc/nginx/sites-enabled/default
sudo nginx -t && sudo nginx -s reload

# 修改后重部署
# 前端：
npm run build:prod && cp -r dist/* /opt/bill-review/dist/
# 后端：
mvn clean package -DskipTests && pkill -f zsc-admin.jar
cp zsc-admin.jar /opt/bill-review/
cd /opt/bill-review && nohup java -jar zsc-admin.jar > logs/app.log 2>&1 &

# ===== 公网 =====
ngrok http 80                        # 启动隧道

# ===== 验证 =====
curl http://127.0.0.1:8080/          # 后端
curl http://127.0.0.1/               # 前端
curl https://xxx.ngrok-free.dev/     # 公网
```
