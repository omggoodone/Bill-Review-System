# 部署说明

## 什么是部署

将开发完成的代码打包，上传到服务器，配置运行环境，让用户通过域名或 IP 访问系统。

## WSL 基础：文件系统与路径

### 为什么需要了解这个

本项目的开发和部署都在一台电脑上完成，依赖 WSL（Windows Subsystem for Linux）同时运行 Windows 和 Ubuntu。理解两边的文件系统关系是部署的前提。

### 两套文件系统

WSL 不是模拟器，它同时存在两套独立的文件系统：

```
Windows 文件系统                    WSL/Linux 文件系统
─────────────────                   ──────────────────
E:\Study\JavaEE\...                  /opt/bill-review/
C:\Users\Lenovo\                     /etc/nginx/
D:\MySQL\...                         /var/lib/mysql/
```

### WSL 如何访问 Windows

WSL 把 Windows 的盘符挂载（mount）到 `/mnt/` 目录下：

```
/mnt/    ← mount（挂载）的缩写
  c/     ← C 盘
  d/     ← D 盘
  e/     ← E 盘
```

| WSL 路径 | 实际位置 |
|----------|----------|
| `/mnt/c/Users/Lenovo/` | `C:\Users\Lenovo\` |
| `/mnt/e/Study/JavaEE/` | `E:\Study\JavaEE\` |

### 本项目两条路径的分工

```
/mnt/e/.../Bill Review System/     ← 源码目录（Windows E 盘，写代码、构建）
         │
         │  cp（拷贝）
         ▼
/opt/bill-review/                  ← 部署目录（WSL 自有文件系统，运行）
```

- **源码放在 Windows**：方便用 IDEA、VS Code 编辑
- **产物拷进 WSL**：`/opt/` 是 Linux 原生文件系统，读写快，适合运行服务

### cp 命令

Linux 下复制文件的命令，对应 Windows 的 Ctrl+C / Ctrl+V。

```bash
cp 源文件 目标位置        # 复制文件
cp -r 源目录 目标位置      # -r：递归复制整个目录

# 实际例子
cp zsc-admin.jar /opt/bill-review/                          # 拷贝 JAR
cp -r /mnt/e/.../zsc_frontend/dist/* /opt/bill-review/dist/ # 拷贝前端构建产物
```

从 `/mnt/e/` 拷到 `/opt/` 就完成了**从 Windows 到 Linux 的文件转移** — 不需要 FTP、不需要 U 盘。

### 为什么部署目录选在 WSL 内部

`/mnt/e/` 下的文件读写要通过 9P 网络协议转一道，I/O 比 Linux 原生文件系统慢很多。所以：

- **构建**在 Windows 侧完成（Maven/npm 本身也是跨平台的）
- **运行**在 WSL 内部 `/opt/`，保证性能

### Linux 关键目录速查

本项目运行时涉及的核心目录：

```
/                          ← 根目录，所有路径的起点
│
├── /opt/                  ← 手动安装的第三方软件
│   └── bill-review/       ← 部署目录
│       ├── dist/          ← 前端静态文件（Nginx 直接返回）
│       ├── logs/          ← 运行日志
│       ├── uploads/       ← 用户上传文件
│       └── zsc-admin.jar  ← 后端可执行 fat JAR
│
├── /etc/                  ← 配置文件集中地（etc = "et cetera" 的缩写）
│   └── nginx/
│       ├── nginx.conf           ← Nginx 主配置
│       └── conf.d/
│           └── bill.conf        ← 本站点配置（来源：deploy/nginx.conf）
│
├── /var/                  ← 经常变化的运行时数据（var = variable）
│   ├── lib/mysql/         ← MySQL 数据库文件
│   └── log/               ← 系统日志
│
├── /mnt/                  ← 外部磁盘挂载点（mnt = mount）
│   ├── c/                 ← Windows C 盘 → C:\Users\...
│   └── e/                 ← Windows E 盘 → E:\Study\JavaEE\...
│
└── /tmp/                  ← 临时文件，重启清空
```

记忆要点：

| 目录 | 含义 | 放什么 |
|------|------|--------|
| `/etc/` | 配置 | Nginx 配置、系统设置 |
| `/var/` | 可变数据 | 数据库、日志 |
| `/opt/` | 可选软件 | 手动部署的应用（本项目） |
| `/mnt/` | 挂载 | Windows 磁盘 |
| `/tmp/` | 临时 | 构建中间产物 |

### /etc 和 /opt 的区别

**这是最容易混淆的两个目录：**

```
/etc/nginx/conf.d/bill.conf     ← 告诉 Nginx "怎么运行"（配置）
/opt/bill-review/zsc-admin.jar  ← 真正的可执行程序（软件本身）
/opt/bill-review/dist/          ← 前端页面文件（软件本身）
```

- `/etc` = **说明书**（配置），告诉程序该怎么干活
- `/opt` = **工具**（程序本身），实际干活的

修改配置去 `/etc`，替换程序去 `/opt`。

## Nginx 是什么

### 一句话

Nginx 是一个**反向代理 + 静态文件服务器**，作为中间人挡在用户和应用程序之间。

### 类比：餐厅的前台

```
没有 Nginx：
  浏览器 ──→ 直接找厨师(Spring Boot :8080)

有 Nginx：
  浏览器 ──→ 前台(Nginx :80) ──→ 厨师(Spring Boot :8080)
```

前台做的事：
- 顾客要菜单（静态文件），前台直接从抽屉拿出来，不惊动厨师
- 顾客要点菜（API 请求），前台转达给厨师
- 前台还能应付更多顾客、排队、安全检查

### 在本项目中的作用

| 功能 | 说明 |
|------|------|
| **静态文件托管** | `/` → 返回 `dist/` 中的 HTML/JS/CSS，相当于高性能 Web 服务器 |
| **API 代理转发** | `/prod-api/` → 转发到 `http://127.0.0.1:8080/`（后端完全不暴露给外部） |
| **Vue History 路由** | `try_files $uri /index.html` — SPA 刷新页面不会404 |
| **Gzip 压缩** | 压缩文本类响应，减少传输体积 |
| **静态资源缓存** | `*.js|css|png` 缓存 30 天 |

### 为什么需要它

- **安全性**: 后端 8080 不对外暴露，只监 listen 127.0.0.1
- **性能**: Nginx 处理静态文件比 Tomcat 快得多
- **反向代理**: 统一入口、负载均衡、SSL 终端、限流都能做
- **职责分离**: Nginx = 高性能门面，Spring Boot = 业务逻辑

## 部署 vs 开发模式

两种运行方式的区别：

| | 部署模式 | 开发模式 |
|---|---|---|
| **前端** | Nginx 托管 `dist/` 静态文件 | Vite dev server 热更新 (`npm run dev`) |
| **后端** | `java -jar` fat JAR | IDE 运行或 `mvn spring-boot:run` |
| **端口** | 统一 `:80`（前端+API同端口） | 前端 `:80`，后端 `:8080` |
| **修改前端** | 改源码 → build → 拷贝 dist | 改完即生效（HMR） |
| **修改后端** | 重新 package → 重启 JAR | 直接重启，可断点调试 |
| **适用场景** | 演示、验收、生产 | 日常开发 |

> 开发阶段建议用**开发模式**，部署模式只用来最终验收。

## 部署架构

```
浏览器 (Windows :80)
    │
    └── WSL Nginx (:80)
           ├─ /              → /opt/bill-review/dist/ （Vue 静态文件）
           └─ /prod-api/*    → 127.0.0.1:8080 （Spring Boot）
                                     │
                                     ├─ MySQL  127.0.0.1:3306  zsc-train
                                     └─ Redis  127.0.0.1:6379
```

**关键点**:
- WSL 网络栈独立，MySQL/Redis 与 Windows 侧的完全隔离
- WSL 端口自动转发到 Windows，所以浏览器 `localhost:80` 能访问 WSL 内的 Nginx

## Nginx 路由规则详解

```nginx
server {
    listen 80;
    server_name localhost;
    root /opt/bill-review/dist;
    index index.html;

    # Vue History 模式 — SPA 路由支持
    # 流程：先找文件 → 找目录 → 都不存在返回 index.html（前端路由接管）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # API 转发 — 核心反向代理规则
    # /prod-api/login → http://127.0.0.1:8080/login
    # 注意 proxy_pass 末尾的 / 会剥离 /prod-api 前缀
    location /prod-api/ {
        proxy_pass http://127.0.0.1:8080/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 60s;
        proxy_read_timeout 60s;
    }

    # 静态资源长期缓存（文件名带 hash，内容变化时文件名也变）
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # 安全：禁止访问 . 开头的隐藏文件（如 .git, .env）
    location ~ /\. {
        deny all;
    }

    # Gzip 压缩
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml text/javascript;
    gzip_min_length 1k;
    gzip_comp_level 6;
}
```

| 路由 | 请求示例 | 实际处理 |
|------|----------|----------|
| `/` | `/login` | → `/opt/bill-review/dist/index.html`（Vue 前端路由接管） |
| `/prod-api/` | `/prod-api/login` | → `http://127.0.0.1:8080/login`（后端 API） |
| `*.js` | `/static/js/app.js` | → 直接返回文件 + 30天缓存头 |
| `/.env` | — | → 403 Forbidden |

### try_files 工作原理

```
用户访问 /login
  → try_files /login (文件不存在)
  → try_files /login/ (目录不存在)
  → 返回 /index.html
  → Vue Router 解析 URL 中的 /login 路径
  → 渲染 Login 组件
```

没有这条规则，刷新页面 `/login` 时 Nginx 找不到对应文件，直接 404。

### proxy_pass 末尾 / 的含义

```nginx
# 有 / — 剥离匹配路径
location /prod-api/ {
    proxy_pass http://127.0.0.1:8080/;
    # /prod-api/login → http://127.0.0.1:8080/login
}

# 无 / — 保留匹配路径
location /prod-api/ {
    proxy_pass http://127.0.0.1:8080;
    # /prod-api/login → http://127.0.0.1:8080/prod-api/login （一般不是你想要的效果）
}
```

### proxy_set_header 的作用

后端需要知道真实客户端信息：

```nginx
proxy_set_header Host              $host;           # 原始域名
proxy_set_header X-Real-IP         $remote_addr;    # 真实客户端 IP（用于日志/审计）
proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;  # 代理链 IP 列表
proxy_set_header X-Forwarded-Proto $scheme;         # 原始协议（http/https）
```

### Nginx 常用命令

```bash
sudo nginx -t              # 检查配置文件语法
sudo nginx -s reload       # 热重载（不中断服务）
sudo nginx -s stop         # 停止
sudo nginx                 # 启动
systemctl status nginx     # 查看运行状态
```

## 本地部署步骤（WSL Ubuntu）

### 前置条件

```bash
# 安装依赖
sudo apt install openjdk-17-jdk maven mysql-server redis-server nginx

# Node.js（通过 nvm）
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
nvm install 18
nvm use 18
```

### 1. 数据库初始化

```bash
# 改 root 认证方式（WSL MySQL 默认 auth_socket → mysql_native_password）
sudo mysql -e "ALTER USER 'root'@'localhost' IDENTIFIED WITH mysql_native_password BY '你的密码'; FLUSH PRIVILEGES;"

# 建库
mysql -u root -p -e "CREATE DATABASE IF NOT EXISTS \`zsc-train\` DEFAULT CHARACTER SET utf8mb4;"

# 导入表结构 + 种子数据
mysql -u root -p zsc-train < zsc_backend/sql/init-system.sql
```

### 2. 构建项目

```bash
# 前端
cd zsc_frontend
npm install
npm run build:prod       # 产物 dist/

# 后端
cd ../zsc_backend
mvn clean package -DskipTests   # 产物 zsc-admin/target/zsc-admin.jar
```

### 3. 部署文件

```bash
sudo mkdir -p /opt/bill-review/{dist,logs,uploads}

sudo cp zsc_frontend/dist/*     /opt/bill-review/dist/
sudo cp zsc-admin.jar            /opt/bill-review/
sudo cp deploy/nginx.conf        /etc/nginx/conf.d/bill.conf
```

### 4. 禁用默认站点 + 启动服务

```bash
# Nginx 默认 site 会抢占80端口，必须移除
sudo rm /etc/nginx/sites-enabled/default
sudo nginx -t && sudo nginx -s reload

# 确保 MySQL、Redis 运行
sudo systemctl start mysql redis-server

# 启动后端
cd /opt/bill-review
nohup java -jar zsc-admin.jar > logs/app.log 2>&1 &
```

### 5. 验证

```bash
curl http://127.0.0.1:8080/    # 后端 → HTTP 200
curl http://127.0.0.1/         # 前端 → 返回 Vue HTML
```

浏览器打开 `http://localhost`，应看到登录页面。

### 6. 首次使用

访问系统 → 登录页显示初始化表单 → 输入邮箱创建超管 → 开始使用。

## 常见问题

### Nginx 显示默认欢迎页

**原因**: 默认站点 `/etc/nginx/sites-enabled/default` 抢占了 80 端口。

**解决**: `sudo rm /etc/nginx/sites-enabled/default && sudo nginx -s reload`

### 修改 nginx.conf 后不生效

```bash
# 必做：先语法检查，再重载
sudo nginx -t
sudo nginx -s reload
```

### 前端刷新页面显示 404

**原因**: 缺少 Vue History 模式的 `try_files` 配置。

**解决**: 确保 `location /` 中有 `try_files $uri $uri/ /index.html;`

### 后端 API 请求返回 404

**原因**: `proxy_pass` 末尾缺少 `/`，导致 `/prod-api` 前缀未被剥离。

**解决**: `proxy_pass http://127.0.0.1:8080/;` 末尾必须有 `/`

### WSL 重启后服务都不在了

WSL 不是真正的 systemd 系统，需要手动启动服务：

```bash
sudo service mysql start
sudo service redis-server start
sudo nginx
cd /opt/bill-review && nohup java -jar zsc-admin.jar > logs/app.log 2>&1 &
```

## 暴露公网：ngrok

### 为什么需要这个

WSL / 家庭网络的 IP 是内网地址（`192.168.x.x` 或 `127.0.0.1`），外网无法直接访问。要让别人通过公网访问你本机上的系统，需要一条**隧道**把公网请求引流到本地。

### 原理

```
公网用户 → https://xxx.ngrok-free.dev
               │
               ▼  ngrok 服务端（云端）
               │   加密隧道
               ▼  ngrok 客户端（你本机，与云端保持长连接）
               │
               ▼  localhost:80 (Nginx)
               │
               ▼  你的应用
```

ngrok 本质上是一个**桥梁**：客户端从你电脑主动连到 ngrok 云端，维持一条长连接。外部用户访问 ngrok 给的域名时，请求通过这条连接"穿透"到你本地。

### 安装

```bash
# Windows 下载解压即可
# https://ngrok.com/download → 下载 Windows 版 → 解压得到 ngrok.exe

# 或者用 winget（Windows 包管理器）
winget install ngrok
```

### 注册与认证

ngrok v3 要求账号认证（免费注册）：

1. 访问 https://dashboard.ngrok.com/signup 注册
2. 登录后获取 authtoken：https://dashboard.ngrok.com/get-started/your-authtoken
3. 配置 token：

```bash
ngrok config add-authtoken <你的token>
```

### 启动

```bash
ngrok http 80
```

成功后显示：

```
Forwarding  https://xxxxxx.ngrok-free.dev  →  http://localhost:80
```

这个 `https://xxxxxx.ngrok-free.dev` 就是公网地址。

### 本地管理面板

启动 ngrok 后访问 `http://localhost:4040`，可以：

- 查看所有进出的请求（URL、状态码、耗时）
- 重放（Replay）某个请求，方便调试
- 查看请求体和响应体

### 免费版限制

| 限制项 | 说明 |
|--------|------|
| 域名变化 | 每次重启 ngrok，域名随机改变 |
| 带宽 | 每月约 1GB 流量 |
| 连接速率 | 约 40 次请求/分钟 |
| 警告页 | 浏览器首次访问会显示 ngrok 警告页，点击 "Visit Site" 即可 |

### 固定域名

免费版也支持申请一个固定的子域名（still free），去 ngrok dashboard 配置即可，这样每次重启域名不变。

### 验证公网可达

```bash
# 命令行测试（需要加跳过警告页的 header）
curl -s -o /dev/null -w "%{http_code}" \
  -H "ngrok-skip-browser-warning: true" \
  https://xxx.ngrok-free.dev/

# 期望输出：200
```

浏览器直接打开的话，第一次点 "Visit Site"，之后就能正常访问。
